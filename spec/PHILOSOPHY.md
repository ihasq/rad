# Rad Philosophy

## Rad は何か

Rad はソースコード管理プロトコルである。Git の哲学的スーパーセットであり、LLM 時代のコード協業基盤として設計された。

Rad は以下を定義する:
- コードの責任がどこに帰属するか
- 変更を受け入れた事実の証明
- 参加者間の序列とその根拠

Rad は以下を定義しない:
- 変更が「正しい」かどうか
- リジェクトの「理由」が妥当かどうか
- 参加者が何者であるか（人間か LLM かは問わない）
- 通信手段（HTTP、P2P、郵便でもよい）

---

## 原則 1: 責任の所在はコードに帰属する

Git はコミットの作者（人間）に責任を帰属させる。Rad はコード領域に責任を帰属させる。

```
Git:  Alice がコミットした → Alice に責任がある（ヒト政治）
Rad:  main.ts:5-10 を最初に書いた者が Leader → その領域の責任を負う（コード政治）
```

人間社会の権限構造（管理者、レビュアー、承認者）はプロトコルの関心外である。誰が CEO であろうと、コード領域に最初に write した者が Leader である。

---

## 原則 2: 受け入れた責任の証明

Rad の核心的責務は「誰が何を受け入れたか」の証明である。

- **accept**: 「この変更を自分のコード領域に受け入れる責任を負う」という署名付き宣言
- **reject**: 「この変更を自分のコード領域に受け入れる責任を負わない」という宣言

reject に理由が必要なのは、Leader が Follower に対して reject する場合のみである。これは「先に責任を負った者が、後から来た者の貢献を拒む場合は説明すべき」という倫理的原則に基づく。逆方向（Follower が Leader のコードを上書きする）は自由であり、改善として歓迎される。

Rad はリジェクトの理由が妥当かどうかを判断しない。「テストが落ちた」も「気に入らない」も同等の文字列である。理由の妥当性を判断するのは参加者（人間社会）の責務であり、プロトコルの責務ではない。

---

## 原則 3: 署名は参加者間で検証される

署名を検証する主体は Relay ではない。Relay は通信を中継するだけであり、内容の正当性を判断しない。

```
誤: Relay が署名を検証し、不正な操作を拒否する
正: Leader が accept する際に、Follower の署名を検証する
    Follower が write する際に、Leader の過去の accept の署名を検証する
    参加者が互いに相手の署名を検証する
```

Relay が署名を検証するのは、中継としてのサービス品質保証（明らかに壊れたデータを転送しない）であり、プロトコル上の責務ではない。Relay が署名検証をスキップしても、プロトコルの正当性は損なわれない。最終的に accept する Leader が検証すればよい。

---

## 原則 4: 全ての write は即座に可視化される

参加者は「今何が起きているか」をリアルタイムに知る権利がある。write は送信された瞬間に全参加者に可視化される（visible 状態）。

これにより:
- 全参加者が他の参加者の作業を認知できる
- 後の参加者は前の参加者の write を見た上で自分の write を判断できる
- Leader は全ての visible な write を認知した上で accept/reject を判断できる

write を隠すことはできない。一度送信された write は全参加者に届く。

---

## 原則 5: コード領域は直列チェーンを形成する

同一コード領域への write は時系列で直列に積み重なる。後の write は前の write を認知した上で書かれている。

```
A → B → C

B を書いた者は A を認知している
C を書いた者は A と B を認知している
```

これは並行する分岐ではない。全ての write が全参加者に即座に可視化されるため、「知らなかった」は成立しない。

Leader が C を accept し B を飛ばす（階段飛ばし）場合、B は消滅する。C の参加者は B が消滅する可能性を認知した上で C を書いた。プロトコルは B の参加者を保護しない。B の参加者が書き直すかどうかは B の参加者の判断である。

---

## 原則 6: Founder はディレクトリ + ファイルの権力チェーン

Founder は唯一の存在ではない。ディレクトリツリーの各階層に Founder が存在し、さらにファイルレベルでも Founder が存在する。これらは直列の権力チェーンを形成する。

```
/ (root Founder: プロジェクトを作成した者)
├── src/ (dir Founder: src/ に最初に write した者)
│   ├── main.ts (file Founder: main.ts を最初に create した者)
│   └── utils.ts (file Founder: utils.ts を最初に create した者)
└── tests/ (dir Founder: tests/ に最初に write した者)
    └── test.ts (file Founder: test.ts を最初に create した者)
```

Founder はディレクトリだけでなくファイルにも存在する。ファイルを最初に create（write で新規作成）した参加者がそのファイルの Founder であり、そのファイルの全コード領域の初期 Leader となる。

ファイル Founder はディレクトリ Founder の Follower である。ディレクトリ Founder はファイル Founder に対して reason なしでリジェクトできる（上位→下位）。

上位ディレクトリの Founder は下位ディレクトリの Founder に対して Leader である。これはコード領域の Leader/Follower モデルがディレクトリ階層に再帰的に適用されたものであり、別の概念ではない。

```
root Founder が src Founder をリジェクト → reason 不要（上位は下位の Leader）
src Founder が root Founder の書き込みをリジェクト → reason 必須（下位は上位の Follower）
dir Founder が file Founder をリジェクト → reason 不要（上位は下位の Leader）
file Founder が dir Founder の書き込みをリジェクト → reason 必須（下位は上位の Follower）
```

root Founder はプロジェクトの基盤（インフラ、ストレージ、CI）を提供した者であり、全ディレクトリの初期 Leader でもある。しかし、あるディレクトリに最初に write した参加者がそのディレクトリの Founder となり、そのスコープ内では Leader として振る舞う。

ファイルの削除（delete）は特別な操作である。delete は「このファイルに対する責任を放棄する」提案であり、ファイルの Founder のみが accept できる。ファイルの Founder 自身が delete を提案した場合、ディレクトリ Founder が accept する。delete は visible 状態で提出され、accept されるまでファイルは削除されない。accept された delete はコンパクション時にファイルを削除する。

Founder の権力はプロトコルが付与する特権ではなく、「そのディレクトリまたはファイルに最初に責任を負った」という事実から生じる。他の参加者が Founder の判断に不同意であれば、コードベースを持ち出して別の基盤に移る自由がある。

---

## 原則 7: トランスポートはプロトコルの関心外

Rad は操作（write, accept, reject）とその署名を定義する。操作がどのように伝達されるかは問わない。

- HTTP Relay 経由で伝達してもよい
- P2P で直接伝達してもよい
- USB メモリで物理的に運んでもよい

Relay は Rad の便利な実装であり、Rad の一部ではない。Rad Relay HTTP API は「Rad プロトコルの HTTP 上のリファレンス実装」であり、Rad プロトコルそのものではない。

---

## 原則 8: 金で買える性能は買えるべき

Git はサーバの性能に上限がある。メモリに載らないリポジトリは処理できない。公開リポジトリをホストするには GitHub のような大規模プロバイダに依存するか、高額なサーバを自前で用意する必要がある。

Rad はエッジコンピューティング基盤（Cloudflare Workers 等）で動作するよう設計される。個々のワーカーは128MB メモリ・低 CPU だが、並列数で性能を購入できる。

```
Git:    金を積んでも動かないものがある
Rad:    金さえ積めばいくらでも速く動く
```

操作ログはインメモリ（Durable Objects）に蓄積し、CRON バッチで安価なストレージ（R2）にコンパクションする。書き込み頻度が高い場合はワーカー数を増やせばよい。

---

## 原則 9: SCM は監査証跡である

Rad が保証するのは「誰が、いつ、何を書き、誰がそれを受け入れたか」の記録の完全性である。

- Operation は Ed25519 で署名され、改竄を検出できる
- accept は Leader の署名付き宣言であり、受け入れた責任の証拠である
- 操作ログは追記専用であり、過去の操作を遡及的に変更できない
- コンパクションは accepted な操作のみを対象とし、visible な操作は保持される

Rad はバージョン管理システムであると同時に、コードの責任に関する監査証跡である。

---

## 原則 10: Git は Rad の「前の参加者」

Rad は Git の哲学的スーパーセットである。Git リポジトリの上に Rad がレイヤー2として乗る。

Git が「前の参加者」であるため、Git のルール（コミット・ブランチ制）が Rad に課される。Rad はこのルールを翻訳し、自身の参加列モデルで再解釈する。

同時に Rad は Git に対して説明義務を負う。Rad の操作は Git が理解できる形式（コミット）に変換され、Git リポジトリに書き戻される。

```
Git → Rad:  Git のコミット履歴を Rad の操作ログとして取り込む（前の参加者のルールを受け入れる）
Rad → Git:  Rad の accepted な操作を Git コミットとして書き出す（後の参加者が説明義務を果たす）
```

将来的に他の SCM（Jujutsu 等）も同様に Rad の「前の参加者」として接続可能である。各 SCM アダプタ（L2）は独立して実装される。
